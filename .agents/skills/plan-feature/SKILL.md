# Plan Feature Workflow Skill

Use this skill when the user wants to start a new feature through the repository's
standard feature-planning flow.

## What this skill does

1. Read `AGENTS.md`, `project.md`, and `.codex/workflows/plan-feature.md`.
2. Follow the interview-first planning flow:
   - always use `$deep-interview` first to clarify boundaries before any `plan.md` drafting;
   - after the interview artifact exists, always use `$ralplan` to converge the executable planning shape;
   - after `deep-interview`, run `node tools/ai-scaffold/dist/cli.js archive-planning-artifacts <feature-dir> --stage deep-interview`;
   - after `$ralplan`, run `node tools/ai-scaffold/dist/cli.js archive-planning-artifacts <feature-dir> --stage ralplan`.
3. Create or update `docs/features/F{nnn}-{slug}/plan.md`.
4. Keep `plan.md` as the only approval artifact of this stage, with planning evidence archived alongside it.

## Required outputs

- `docs/features/F{nnn}-{slug}/plan.md`
- `docs/features/F{nnn}-{slug}/reports/planning/deep-interview.md`
- `docs/features/F{nnn}-{slug}/reports/planning/prd.md`
- `docs/features/F{nnn}-{slug}/reports/planning/test-spec.md`
- Required OMX runtime artifacts:
  - `.omx/context/*`
  - `.omx/interviews/*`
  - `.omx/specs/deep-interview-*.md`
  - `.omx/plans/prd-*.md`
  - `.omx/plans/test-spec-*.md`

## Hard rules

- Do not implement feature code in this stage.
- Do not draft, update, or finalize any `plan.md` before a `deep-interview` clarification pass has been completed for that feature.
- Do not skip `$ralplan` after `deep-interview`; `plan.md` must be based on the resulting PRD / test-spec outputs.
- `plan.md` must stay `plan_status: draft` until a human changes it to `approved`.
- `plan.md` must include `Intent`, `Out of Scope / Non-goals`,
  `Decision Boundaries`, and `Exception Scenarios`.
- Planning evidence generated by `deep-interview` and `$ralplan` must be archived under the same feature directory in `reports/planning/`.
- Prefer the scaffold command above instead of manual copy/paste when archiving planning evidence.
- Use `docs/features/NEXT_FEATURE_NUMBER.txt` when a new feature number is needed.

## Invocation

- Explicit: `$plan-feature <feature-dir|slug|idea>`
- Equivalent repository command: `/plan-feature <feature-dir|slug|idea>`

## Execution note

This skill is an execution surface for `.codex/workflows/plan-feature.md`.
When invoked, read that workflow and carry it out rather than improvising a new flow.
