﻿# PRD: F008-dataset-preparation-pipeline

## Problem

当前平台虽已具备数据资产、训练任务与模型管理的前置能力，但算法/训练团队仍缺少一个平台内自助完成训练前数据准备的正式入口。若继续依赖分散脚本、人工处理或外部工具，将导致数据来源、清洗规则、标注口径、划分方式、预处理参数、增强策略和格式转换结果难以统一、复用、审计与重跑。

## Goals

- 建立平台内置的数据准备流水线，覆盖数据收集、清洗、标注、划分、预处理、增强、格式转换与加载配置 7 个核心步骤。
- 支持多来源原始数据统一接入，并沉淀为可直接用于训练的标准数据集。
- 对全过程保留配置、质量检查结果、数据版本、重跑记录与权限隔离，支持多人协作。
- 明确失败阻断与人工修正重跑机制，保证标准训练数据集的质量门禁。
- 在现有数据资产与权限体系基础上扩展能力，避免平行重复实现。

## Users

- 算法工程师：自助完成训练前数据准备并获得可直接用于训练的数据集。
- 数据工程师 / 数据管理员：配置采集、清洗、转换与版本规则，处理异常并重跑。
- 标注人员：执行或复核标注任务，确保标签一致性与准确性。
- 平台管理员 / 审计人员：查看配置、过程、质量结果、权限边界与重跑记录。

## Functional Requirements

- 支持从公开数据集、企业内部系统、网络采集等来源创建数据准备任务并登记来源元数据。
- 支持数据清洗规则配置与执行，包括去噪、异常值处理、重复样本识别、缺失值处理、格式统一。
- 支持监督学习场景的数据标注流程、标签结构定义、标注一致性与准确性检查。
- 支持训练集 / 验证集 / 测试集划分，允许配置比例并保留划分快照。
- 支持多模态预处理能力的统一编排：
  - 图像：归一化、裁剪、翻转、旋转等；
  - 文本：分词、编码、token 序列化；
  - 其他模态在计划中预留相同流程位点。
- 支持数据增强配置与执行，生成新的样本版本并记录增强策略。
- 支持将处理结果转换为训练框架所需格式，并生成加载配置 / DataLoader 元数据。
- 支持阶段状态、质量门禁、失败原因、重跑入口与审计记录。
- 支持多人协作、权限隔离与关键操作留痕。

## Non-goals

- 训练任务提交、资源调度、训练执行监控。
- 模型注册、审批、发布、治理与推理部署。
- 将任一核心步骤委托给平台外脚本作为正式交付路径。
- 首期绑定未确认的真实企业外部系统清单或生产合规策略。

## Product Decisions

- 首期即按“完整训练前数据准备流水线”规划，不拆成只做部分步骤的轻量版。
- 所有 7 个核心步骤均要求平台内置能力承接，不将关键能力外包给外部系统步骤。
- 默认异常策略为“任一步骤失败或质量门禁不达标即阻断，人工修正后重跑”。
- F008 产物边界止于“可直接用于训练的标准数据集”，不跨入训练与模型治理链路。
- F008 应建立在 F004 数据资产域之上扩展为数据准备流水线，而不是新建第二套数据域。

## Reuse Strategy

### Must Reuse

- `backend/src/main/java/com/yfind/aiplatform/dataset/*`：现有数据集、版本、处理任务、事件与控制器骨架。
- `backend/src/main/java/com/yfind/aiplatform/permission/*`、`identity/*`：现有权限、会话、审批和审计基线。
- `backend/src/main/java/com/yfind/aiplatform/training/*`：格式转换结果与训练侧衔接时的数据引用模型边界。
- `frontend/src/pages/DatasetPage.tsx`、`frontend/src/modals/UploadDatasetModal.tsx`、`frontend/src/api/datasetApi.ts`：现有数据资产页面和交互入口。
- `frontend/src/pages/TrainingPage.tsx`、`frontend/src/api/trainingApi.ts`：用于界定 F008 与训练侧的边界，不重复创建训练配置面。
- `tools/ai-scaffold`、`docs/features/` 工作流、既有 F004/F005 报告结构与测试组织方式。

### Duplication Rejected

- 不创建第二个独立“数据准备服务”绕开现有 dataset domain。
- 不复制第二套权限 / 审批 / 审计机制。
- 不在 `ai-adapter/` 或其他新目录重复实现数据准备主业务流程。
- 不为不同来源或不同模态各自创建平行的独立流水线框架。

### Approved New Seams

- 新增“数据准备任务 / 阶段 / 规则快照 / 质量报告 / 重跑记录 / 标注任务”等领域对象，因为现有 dataset domain 尚未覆盖完整流程编排。
- 新增多阶段流水线状态机与步骤执行 seam，因为 F004 仅具备数据资产管理，不具备全流程数据准备编排。
- 新增面向不同模态的预处理与增强策略抽象，因为现有仓库缺少统一的可配置处理层。

## Success Criteria

- `plan.md`、`prd.md`、`test-spec.md` 明确 7 个步骤全量纳入 F008 首期范围。
- 后续 build-feature 可围绕“采集 -> 清洗 -> 标注 -> 划分 -> 预处理 -> 增强 -> 转换/加载”形成可验证的端到端流程。
- 所有阶段的配置、质量检查结果、版本、失败原因与重跑记录可追溯。
- 权限隔离与多人协作边界在 contract/test-plan 中被正式冻结。
- F008 与训练任务 / 模型治理的边界清晰，不产生职责重叠。

## Risks

- 一次性纳入 7 个完整步骤会显著扩大领域模型与前后端交互复杂度。
- 多模态处理（图像 / 文本 / 其他）若在首期都要求同等深度，可能导致范围失衡。
- 网络采集与企业内部系统接入存在合规、权限与外部依赖不确定性。
- 标注一致性校验、质量门禁和失败阻断策略若定义不清，容易造成流程卡死或误放行。
- 若复用边界处理不当，F008 可能和 F004 数据资产能力出现重复建设。

## Open Questions

- 首期必须优先保证的模态是图像、文本，还是两者并行同级？
- 企业内部系统接入首期是否只做可配置连接抽象，不绑定真实系统？
- 网络采集能力首期是否仅支持受控来源登记，而不开放通用爬虫策略？
- 标注首期是否包含人工标注工作台，还是先包含标签结构、任务流转和结果接入能力？
- 格式转换首期要优先支持哪些训练格式（例如通用目录结构、TFRecord、文本 token 包等）？
