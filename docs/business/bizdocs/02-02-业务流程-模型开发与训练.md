# 02-02 业务流程清单 – 模型开发与训练域

> **范围**：模型开发与训练域（MODEL）包含模型调优、模型评估、模型管理、模型工程化、模型开发环境及训练任务等完整的模型生命周期管理流程。
> **组织方式**：按子流程分组，每个流程含触发条件、主流程步骤、分支/异常处理。

---

## 一、模型调优流程（MODEL-001）

**触发条件**：模型训练工程师选择预训练模型进行场景适配

**主要参与角色**：
- 数据标注工程师
- 模型训练工程师
- BU级子管理员
- 超级管理员

**主流程步骤**：

1. 选择基础预训练模型 → 从模型库中选择合适的预训练模型
2. 选择标注数据集 → 从数据集门户选择相关的标注数据集
3. 数据集拆分 → 划分训练集、验证集、测试集（通常按8:1:1或7:2:1比例）
4. 配置超参数 → 
   - 手动配置：学习率、批次大小、迭代轮数、优化器等
   - 自动超参数调优：启用AutoML功能自动搜索最优超参数
5. 配置资源 → 申请计算资源（CPU/GPU/NPU），指定资源数量和类型
6. 提交调优任务 → 将调优任务提交到任务队列
7. 监控训练过程 → 
   - 实时查看训练日志
   - 监控资源使用情况（CPU、内存、GPU显存）
   - 在TensorBoard中观察训练曲线和指标变化
8. 评估验证 → 在验证集上评估模型性能
9. 导出调优产物 → 保存训练完成的模型或推送到边端服务器

**分支流程**：

- **自动超参数调优（AutoML）**：
  - 定义超参数搜索空间
  - 选择搜索算法（网格搜索、随机搜索、贝叶斯优化）
  - 自动执行多轮超参数组合实验
  - 返回最优超参数组合

- **分布式训练**：
  - 多节点并行训练
  - 配置分布式策略（数据并行、模型并行）
  - 跨机器同步梯度
  - 加速训练过程

**异常流程**：
- 资源不足：进入等待队列，或申请扩容后优先调度
- 训练发散：调整学习率、批次大小等超参数后重新训练
- GPU内存溢出：减小批次大小或启用梯度累积
- 训练中断：支持断点续训

**主要输入**：预训练模型、标注数据集、超参数配置、资源申请

**主要输出**：调优后的模型、训练日志、性能评估报告

**关联功能**：FUNC-MODEL-001, FUNC-MODEL-002, FUNC-MODEL-003

## 二、模型评估流程（MODEL-002）

**触发条件**：模型训练完成或需要对现有模型进行性能评估

**主要参与角色**：
- 数据标注工程师
- 模型训练工程师
- BU级子管理员
- 超级管理员

**主流程步骤**：

1. 选择待评估模型 → 从模型库中选择目标模型
2. 选择验证数据集 → 选择独立的验证数据集（未在训练中使用）
3. 配置评估指标 → 根据任务类型选择合适的评估指标：
   - **分类任务**：准确率（Accuracy）、精准率（Precision）、召回率（Recall）、F1分数、ROC-AUC
   - **目标检测**：mAP（平均精准率）、IoU（交并比）
   - **语义分割**：Dice系数、mIoU（平均交并比）
   - **NLP任务**：BLEU、ROUGE、困惑度（Perplexity）
   - **时间序列预测**：MAE、RMSE、MAPE
4. 执行评估 → 在验证数据集上运行模型推理并计算指标
5. 查看多维度评估报告 → 
   - 整体性能指标
   - 按数据子集的性能分析
   - 错误案例分析
   - 混淆矩阵（分类任务）
6. 决策 → 
   - 通过：模型性能满足要求，可推进到部署
   - 继续调优：性能未达预期，调整超参数或数据后重新训练

**异常流程**：
- 评估指标选择不当：咨询领域专家，重新配置评估指标
- 验证数据集数据泄露：重新选择独立的验证数据集

**主要输入**：待评估模型、验证数据集、评估指标配置

**主要输出**：评估报告、性能指标、错误分析

**关联功能**：FUNC-MODEL-004, FUNC-MODEL-005

## 三、模型管理流程（MODEL-003）

**触发条件**：模型训练完成需要纳管，或用户上传本地模型

**主要参与角色**：
- 数据标注工程师
- 模型训练工程师
- BU级子管理员
- 超级管理员

**主流程步骤**：

### 场景A：从训练中心发布模型

1. 训练完成 → 模型训练任务完成
2. 提交模型 → 选择待发布的模型文件
3. 填写元数据 → 
   - 模型名称
   - 模型描述
   - 版本号
   - 创建日期
   - 模型所有者
   - 关联的数据集和训练配置
4. 选择模型类型 → 
   - 平台级模型：所有用户可见和使用
   - 团队级模型：仅团队内成员可见和使用
5. 发布版本 → 确认发布，模型进入可用状态

### 场景B：用户上传本地模型

1. 上传模型文件 → 上传本地训练的模型文件（支持多种格式）
2. 填写元数据 → 完整描述模型信息
3. 权限配置 → 设置模型访问权限（公开/团队内/个人）
4. 版本管理 → 创建初始版本记录
5. 激活使用 → 模型可被其他用户引用

### 版本管理

1. 创建版本 → 为同一模型创建不同版本
2. 标记状态 → 为每个版本标记生命周期状态：
   - 开发（Development）：模型开发阶段
   - 测试（Testing）：模型测试阶段
   - 生产（Production）：已上线生产环境
   - 废弃（Deprecated）：不再维护的版本
3. 版本对比 → 查看不同版本的性能差异和参数变化

**异常流程**：
- 模型文件损坏：重新上传或从训练中心重新生成
- 版本冲突：通过时间戳和版本号进行版本管理

**主要输入**：训练完成的模型或本地模型文件、元数据

**主要输出**：已发布的模型、模型版本记录、权限配置

**关联功能**：FUNC-MODEL-006, FUNC-MODEL-007, FUNC-MODEL-008

## 四、模型工程化流程（MODEL-004）

**触发条件**：模型完成开发和验证，需要进行优化以便部署到边端或生产环境

**主要参与角色**：
- 数据标注工程师
- 模型训练工程师
- BU级子管理员
- 超级管理员

**主流程步骤**：

1. 选择待工程化模型 → 从模型库中选择目标模型版本
2. 选择优化方式 → 根据部署需求选择一种或多种优化方法：
   - **量化**：降低模型精度（如FP32→INT8）以减小模型体积和推理时间
   - **剪枝**：移除不重要的权重和通道以减小模型大小
   - **蒸馏**：用小模型学习大模型的行为以保持性能同时减小模型
3. 配置目标硬件 → 指定部署目标硬件平台：
   - CPU（x86、ARM）
   - GPU（NVIDIA、AMD、Intel）
   - NPU（华为NPU、高通AI Engine）
   - 边端设备（边缘服务器、物联网设备）
4. 执行优化 → 触发模型优化任务
5. 跨平台转换 → 将优化后的模型转换为目标格式：
   - ONNX：开放神经网络交换格式
   - TensorRT：NVIDIA推理优化框架
   - TFLite：TensorFlow Lite移动部署格式
   - OpenVINO：英特尔推理优化框架
   - 专有格式（根据硬件厂商支持）
6. 验证性能 → 
   - 比对优化前后的推理速度
   - 比对优化前后的模型精度
   - 验证在目标硬件上的兼容性
7. 导出或推送 → 
   - 导出优化后的模型文件
   - 推送到边端服务器进行部署

**异常流程**：
- 优化后精度下降过大：调整优化参数或选择其他优化方式
- 目标硬件不兼容：更换目标硬件或调整转换格式
- 推理时间未达预期：尝试更激进的优化策略

**主要输入**：模型文件、优化配置、目标硬件信息

**主要输出**：优化后的模型、格式转换文件、性能对比报告

**关联功能**：FUNC-MODEL-009, FUNC-MODEL-010

## 五、模型开发环境流程（MODEL-005）

**触发条件**：模型训练工程师需要进行模型开发、训练或调试

**主要参与角色**：
- 数据标注工程师
- 模型训练工程师
- BU级子管理员
- 超级管理员

**主流程步骤**：

1. 创建开发任务 → 新建一个模型开发任务
2. 选择开发镜像 → 选择包含所需框架的Docker镜像：
   - **PyTorch**：深度学习通用框架
   - **TensorFlow**：谷歌深度学习框架
   - **PaddlePaddle**：百度深度学习框架
   - **其他框架**：MXNet、Theano等
3. 申请资源 → 配置开发环境所需资源：
   - CPU核数
   - 内存大小（GB）
   - GPU数量和型号（可选）
4. 挂载数据集与模型 → 
   - 挂载训练数据集
   - 挂载预训练模型
   - 挂载代码库
5. 启动开发环境 → 选择开发工具：
   - **Jupyter Notebook**：交互式编程环境
   - **VSCode**：代码编辑和调试
   - **SSH**：远程命令行访问
6. 开发调试 → 工程师在开发环境中编写和调试代码
7. 调用推理服务 → 利用已部署的推理服务进行模型推断和辅助编程
8. 发布产物 → 完成开发后选择发布方式：
   - 发布模型到模型中心：将训练完成的模型保存到模型库
   - 制作镜像发布到镜像中心：将开发环境打包为镜像供后续使用

**异常流程**：
- 资源配额超限：申请扩容或等待资源释放
- 环境启动失败：检查镜像配置和资源配置后重试
- 数据挂载失败：检查数据集权限和存储连接

**主要输入**：开发镜像、资源配置、数据集、预训练模型

**主要输出**：开发环境访问凭证、训练模型、开发镜像

**关联功能**：FUNC-MODEL-011, FUNC-MODEL-012, FUNC-MODEL-013

## 六、训练任务流程（MODEL-006）

**触发条件**：需要提交模型训练任务（单机或分布式）

**主要参与角色**：
- 数据标注工程师
- 模型训练工程师
- BU级子管理员
- 超级管理员

**主流程步骤**：

1. 创建训练任务 → 新建训练任务，选择训练类型：
   - 单机训练：在单个计算节点上进行训练
   - 分布式训练：在多个计算节点上并行训练
2. 选择镜像 → 选择包含训练框架和依赖的Docker镜像
3. 配置训练参数 → 
   - 学习率、批次大小、迭代轮数等超参数
   - 数据加载配置
   - 模型保存间隔
   - 早停策略等
4. 配置启动指令 → 指定训练脚本和启动命令（如：python train.py --lr=0.001）
5. 申请资源 → 指定所需计算资源：
   - CPU核数
   - 内存大小
   - GPU数量和型号（可选）
   - 分布式训练需指定节点数
6. 挂载数据集 → 将训练数据集挂载到训练容器
7. 提交任务 → 将训练任务提交到任务队列
8. 监控训练 → 
   - 实时查看训练日志输出
   - 使用TensorBoard可视化训练过程
   - 监控计算资源使用情况（CPU、内存、GPU利用率）
   - 监控网络流量（分布式训练）
9. 任务完成 → 训练任务完成或达到停止条件
10. 发布训练产物 → 将训练完成的模型保存为可复用的模型版本

**分支流程**：

- **分布式训练**：
  - 配置分布式框架（Horovod、PyTorch DDP、TensorFlow MultiWorker等）
  - 设置通讯后端（NCCL、Gloo、MPI）
  - 配置数据并行或模型并行策略
  - 监控节点间通讯状态

- **多GPU单机训练**：
  - 自动分配多个GPU
  - 启用数据并行加速
  - 监控单机内GPU间通讯

**异常流程**：
- 任务提交失败：检查资源配置和镜像可用性
- 训练过程中GPU显存溢出：减小批次大小或启用梯度检查点
- 节点间通讯出现故障：检查网络连接，重新提交任务
- 训练中断：支持从保存点恢复训练

**主要输入**：训练脚本、镜像、数据集、超参数配置、资源申请

**主要输出**：训练完成的模型、训练日志、性能指标

**关联功能**：FUNC-MODEL-014, FUNC-MODEL-015, FUNC-MODEL-016
