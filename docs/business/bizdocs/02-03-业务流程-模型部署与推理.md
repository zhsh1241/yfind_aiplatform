# 02-03 业务流程清单 – 模型部署与推理域

> **范围**：模型部署与推理域（INF）包含中心端推理服务部署、第三方推理服务纳管和边端服务器模型下发等完整的推理服务部署流程。
> **组织方式**：按子流程分组，每个流程含触发条件、主流程步骤、分支/异常处理。

---

## 一、中心端推理服务部署流程（INF-001）

**触发条件**：模型完成评估和优化，需要部署为在线推理服务

**主要参与角色**：
- 数据标注工程师
- 模型训练工程师
- 模型应用工程师
- BU级子管理员
- 超级管理员

**主流程步骤**：

1. 选择待部署模型版本 → 从模型库中选择生产就绪的模型版本
2. 配置服务参数 → 
   - **通讯协议选择**：
     - HTTP/REST：简单易用，适合通用场景
     - WebSocket：支持实时双向通讯，适合流式推理
     - gRPC：高性能协议，适合高并发场景
   - **资源配额配置**：
     - CPU分配（核数）
     - 内存分配（GB）
     - GPU配置（数量、型号）
   - **扩缩容策略**：
     - 最小实例数
     - 最大实例数
     - 自动扩缩容触发条件（CPU利用率、内存利用率、请求延迟）
3. 配置安全围栏 → 
   - 启用内容过滤模块
   - 配置敏感内容识别规则
   - 设置输出安全检测
4. 配置流量限制 → 
   - 设置单用户QPS限制
   - 设置全局QPS限制
   - 设置请求超时时间
   - 配置限流策略（令牌桶/滑动窗口）
5. 手动部署上线 → 触发服务部署流程
6. 拨测验证 → 
   - 端口探活：检查服务端口是否开放和响应
   - 服务验证：发送测试请求验证服务功能正确
   - 性能基准测试：测试推理延迟和吞吐量
7. 服务上线 → 服务通过验证，正式对外提供服务
8. 监控（24h持续） → 
   - 监控服务调用量（QPS、请求数）
   - 监控计算资源使用情况（CPU、内存、GPU）
   - 监控服务状态和可用性
   - 监控错误率和异常请求
   - 设置告警规则

**分支流程**：

- **在线实时推理服务**：
  - 小批量实时请求处理
  - 低延迟响应（毫秒级）
  - 高并发支持
  - 自动扩缩容以应对突发流量

- **批量推理离线处理**：
  - 大规模数据批处理
  - 可预定时间执行
  - 支持设置优先级队列
  - 支持定期定时执行

**异常流程**：
- 服务不可用（宕机或重启）：
  - 立即查看服务日志诊断原因
  - 进入服务容器终端进行深度调试
  - 根据问题类型采取恢复措施（重启、回滚等）
- 无请求时自动缩容到0：节省资源成本，请求到达时自动扩容

**主要输入**：模型版本、服务配置、安全策略、资源需求

**主要输出**：推理服务URL、服务监控面板、告警配置

**关联功能**：FUNC-INF-001, FUNC-INF-002, FUNC-INF-003

## 二、第三方推理服务纳管流程（INF-001b）

> ⚠️ **范围说明**：本流程暂不在一期交付范围内，后续版本再行规划。

**触发条件**：需要集成和管理第三方AI服务（如云厂商推理服务、合作伙伴服务）

**主要参与角色**：
- 运维工程师
- 推理服务管理员

**主流程步骤**：

1. 填写第三方服务信息 → 
   - 服务名称
   - 服务URL或接入地址
   - 调用协议（HTTP/gRPC等）
   - 认证方式（API Key、OAuth、证书等）
   - 服务提供商名称
   - 服务能力描述
2. 连接测试 → 
   - 使用提供的认证信息连接第三方服务
   - 发送测试请求验证连接可用性
   - 测试服务响应时间和稳定性
3. 配置访问权限 → 
   - 设置哪些用户/团队可以使用该服务
   - 配置权限级别（读/写/管理）
4. 配置时长 → 
   - 设置服务可用期限
   - 配置自动续期策略
5. 纳管激活 → 第三方服务被平台纳管，可供其他流程引用

**异常流程**：
- 连接失败：检查URL、认证信息、网络连通性
- 权限被拒绝：检查API Key是否有效，联系服务提供商
- 服务不稳定：联系服务提供商，配置备用服务

**主要输入**：第三方服务信息、认证凭证

**主要输出**：已纳管的第三方服务、集成配置

**关联功能**：FUNC-INF-004, FUNC-INF-005

## 三、边端服务器管理与模型下发流程（INF-002）

**触发条件**：需要在边端部署模型以支持本地推理，降低网络延迟

> **一期范围说明**：只考虑对边缘端服务器进行统一纳管，支持模型快速下发、版本管理即可；不考虑远程监控、故障诊断和升级管理。

**主要参与角色**：
- 模型应用工程师
- BU级子管理员
- 超级管理员

**主流程步骤**：

### 阶段一：边端服务器注册与纳管

1. 注册边端服务器 → 
   - 填写服务器信息：
     - 服务器名称（如：工厂A车间-推理服务器-001）
     - IP地址/主机名
     - SSH端口
     - 操作系统类型（Linux/Windows）
     - 硬件配置（CPU型号、核数、内存、GPU等）
   - 配置访问凭证：
     - SSH用户名和密码（或证书）
     - 连接超时设置
2. 连接测试 → 
   - 测试与边端服务器的网络连通性
   - 验证SSH连接可用性
   - 检查存储空间是否充足
   - 检查依赖环境（Docker、运行时）是否已安装
3. 统一纳管 → 服务器通过验证，纳入平台管理，可进行后续部署操作

### 阶段二：模型选择与配置

4. 选择模型 → 选择待下发模型：
   - 已调优的模型：直接完成的模型版本
   - 已工程化的模型：经过量化、剪枝等优化的模型
5. 配置下发策略 → 
   - **下发时机**：
     - 立即下发：现在立即推送到边端服务器
     - 定时下发：在指定时间推送（如非工作时间）
   - **更新策略**：
     - 全量更新：替换现有模型
     - 灰度更新：先在部分设备更新验证
     - 金丝雀更新：限制版本影响范围
   - **回退策略**：
     - 新版本出现问题时自动回退到上一版本
     - 配置回退触发条件（错误率、推理失败等）

### 阶段三：模型下发与验证

6. 执行模型下发 → 
   - 压缩模型文件
   - 计算文件校验和
   - 推送到边端服务器
   - 在边端解压和安装模型
   - 验证文件完整性
7. 验证边端服务可用性 → 
   - 检查模型加载是否成功
   - 发送测试推理请求
   - 验证推理结果准确性
   - 检查推理延迟是否满足要求

### 阶段四：边端应用开发

8. 边端应用开发 → 开发人员基于已部署的模型进行应用开发：
   - 集成本地推理SDK
   - 开发业务逻辑和界面
   - 处理离线场景（网络不可用时的本地推理）
   - 边端数据采集和预处理

**分支流程**：

- **多个边端服务器管理**：
  - 创建服务器组进行批量管理
  - 配置相同模型到多个服务器
  - 统一监控所有边端服务器状态

- **模型版本灰度更新**：
  - 选择小部分边端先行更新
  - 监控新版本运行情况
  - 确认无问题后推广到全部边端

**异常流程**：
- 连接失败：检查网络连通性、SSH配置、防火墙规则
- 模型下发失败：检查存储空间、网络带宽、文件完整性
- 推理验证失败：检查模型兼容性、依赖库版本、硬件支持
- 自动回退：版本出现严重错误时触发上一版本的自动回退

**主要输入**：边端服务器信息、模型版本、下发策略配置

**主要输出**：模型部署状态报告、推理验证结果、部署日志

**关联功能**：FUNC-INF-006, FUNC-INF-007, FUNC-INF-008

## 四、推理服务调用与集成（横向关联）

**说明**：各业务流程中对推理服务的调用遵循以下标准：

1. **API调用方式**：
   - REST API：HTTP GET/POST请求，请求体为JSON格式
   - gRPC：高性能二进制调用，支持流式传输
   - WebSocket：实时双向通讯

2. **请求/响应格式**：
   - 所有请求需包含用户认证信息
   - 所有响应包含状态码和错误信息
   - 推理结果使用统一的JSON Schema

3. **错误处理**：
   - 网络错误：自动重试（指数退避）
   - 服务超时：返回超时错误，不重试长期请求
   - 服务不可用：返回503，引导用户等待或使用备用服务

4. **性能优化**：
   - 支持批量推理以提高吞吐量
   - 支持异步推理以降低延迟影响
   - 启用结果缓存以减少重复计算
