## 规则编码规范

每条规则采用 `{类型缩写}-{三位序号}` 格式编码，例如：
- `DAT-001`：数据管理规则 1——数据源连接测试必须通过方可激活
- `MDL-002`：模型开发规则 2——训练任务提交前须通过 GPU/NPU 资源配额检查
- `INF-006`：推理部署规则 6——新版本健康检查失败时自动回滚至上一版本
- `RES-004`：资源管理规则 4——基础镜像须通过安全扫描后方可用于训练和推理
- `PLT-004`：平台权限规则 4——连续登录失败 5 次后账号锁定 30 分钟

---

## 规则严重等级

| 等级 | 含义 | 处理方式 |
|------|------|---------|
| **MUST** | 硬性卡控，违反则阻断操作 | 必须 pass，fail 则整条流程判定失败 |
| **SHOULD** | 软性校验，违反则警告但可继续 | 扣分项，展示警告，不阻断流程 |
| **CONFIG** | 可配置规则，按 BU / 角色 / 租户维度开关 | 根据配置决定是否执行校验 |

## 规则文档结构约定

每个规则文件采用统一结构：

```
### {规则编码} {规则名称}
- **等级**: MUST / SHOULD / CONFIG
- **触发场景**: 何时触发此规则
- **规则描述**: 业务逻辑的精确定义
- **校验逻辑**: 系统如何校验
- **异常处理**: 校验失败时的系统行为
- **配置项**: 涉及的主数据配置（如有）
- **交互终端**: PC / APP / Interface / 系统自动
```
