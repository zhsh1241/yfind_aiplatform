# YFI 工业 AI 小模型平台项目计划

## 1. 计划假设

| 项目 | 假设 |
| --- | --- |
| 周期 | 6 个月 MVP |
| 团队 | 中型团队，约 8-12 人 |
| 部署 | 私有化 Kubernetes |
| 建设方式 | 成熟组件集成为主，自研门户、权限、业务编排和治理层 |
| MVP 范围 | 先完成一条工业 AI 小模型平台闭环 |
| 实施方法 | PoC 验证 -> 架构基线 -> MVP 迭代 -> UAT -> 上线 |

MVP 验收主线：

```text
一个 BU 用户登录平台 -> 上传/登记数据集 -> 创建标注任务 -> 审核标注数据 -> 提交训练任务 -> 注册模型版本 -> 部署中心端推理服务 -> 下发到边端服务器 -> 查看资源、日志和审计
```

## 3. 阶段计划

| 阶段 | 周期 | 目标 | 主要交付物 | 验收标准 | 依赖与风险 |
| --- | --- | --- | --- | --- | --- |
| 第 0 阶段：需求澄清与 PoC 准备 | 2 周 | 锁定 MVP 范围、环境、关键组件和验收主线 | 需求确认清单、MVP backlog、PoC 计划、环境清单 | SOR 范围完成评审，待确认项有 owner 和截止时间 | SSO、K8s、GPU/NPU、存储、边端条件可能不完整 |
| 第 1 阶段：架构基线与工程底座 | 4 周 | 建立可运行工程骨架、权限底座、部署底座和组件 PoC | 后端/前端工程、CI/CD、认证 PoC、对象存储 PoC、K8s 部署基线 | 用户可登录，基础菜单可见，后端/前端/部署流水线可运行 | 企业 IAM 和私有化环境需提前可用 |
| 第 2 阶段：数据资产与标注 MVP | 6 周 | 完成数据源、数据集、对象存储、标注和审核闭环 | 数据集管理、版本、权限申请、Label Studio 集成、审核流 | 可上传数据集、创建标注任务、回收标注结果并生成标注数据集版本 | 标注工具集成深度和数据格式需控制范围 |
| 第 3 阶段：训练、模型注册、评估 MVP | 6 周 | 完成训练任务、调优任务、日志指标、模型注册和评估 | 训练任务、Kubeflow/Argo 集成、MLflow 集成、模型中心、评估报告 | 可基于标注数据集启动训练，生成模型版本和评估指标 | GPU/NPU 镜像、训练样例和资源配额需稳定 |
| 第 4 阶段：推理服务与边端下发 MVP | 4 周 | 完成中心端推理服务部署和边端服务器模型下发 | KServe 部署、推理服务管理、边端 agent、模型包下发、状态回传 | 模型可部署为推理服务并完成一次边端下发 | 边端网络、操作系统和安全策略需提前验证 |
| 第 5 阶段：资源、监控、安全、审计补强 | 4 周 | 补齐资源池、监控、审计、安全和运维能力 | 资源配额、基础监控、日志查询、审计查询、安全基线 | 关键操作有审计，资源和服务有监控，异常可定位 | 安全策略和审计留存要求需业务确认 |
| 第 6 阶段：UAT、性能测试、上线准备 | 2 周 | 完成 UAT、性能基线、缺陷收敛和上线材料 | UAT 报告、性能报告、部署手册、运维手册、上线 checklist | MVP 主线通过 UAT，P0/P1 缺陷清零，上线方案评审通过 | 真实数据、业务用户和生产环境窗口需协调 |

## 4. 里程碑

| 里程碑 | 时间点 | 通过标准 |
| --- | --- | --- |
| M0：SOR 范围确认 | 第 2 周末 | MVP 范围、延期范围、验收主线和待确认事项完成签字确认 |
| M1：技术 PoC 通过 | 第 6 周末 | SSO、对象存储、Label Studio、训练、MLflow、KServe、边端 agent 至少完成核心验证 |
| M2：工程底座可运行 | 第 6 周末 | 后端、前端、部署、CI/CD、权限和基础审计可运行 |
| M3：数据集与标注闭环可用 | 第 12 周末 | 数据集上传、标注、审核、版本化和权限申请闭环可演示 |
| M4：训练到模型注册闭环可用 | 第 18 周末 | 训练任务可运行，指标和模型版本进入模型中心 |
| M5：模型部署与边端下发闭环可用 | 第 22 周末 | 模型可部署为服务，并可下发到边端服务器 |
| M6：MVP UAT 通过 | 第 26 周末 | 业务验收通过，上线材料和运维交接完成 |

## 5. 第一批 Feature Backlog

| Feature | 目标 | 主要验收标准 |
| --- | --- | --- |
| F001-platform-architecture-baseline | 建立项目架构、工程目录、部署基线和组件版本矩阵 | 后端/前端/部署骨架可运行，技术 PoC 计划明确 |
| F002-identity-org-permission | 实现登录、组织、用户、角色、RBAC/ABAC 基线和审计 | 用户按角色看到不同菜单和资源，越权访问被拒绝 |

| F004-labeling-workflow-mvp | 集成标注工具，实现任务创建、分配、回收和审核 | 标注任务可完成并生成可训练的数据集版本 |
| F005-training-job-mvp | 实现训练任务提交、资源申请、日志、指标和产物 | 训练任务可在 K8s 运行并生成 artifact |
| F006-model-registry-evaluation | 实现模型注册、版本、评估指标和模型权限 | 模型版本可查询、评估、授权和导出 |
| F007-inference-service-mvp | 实现 KServe 推理服务部署、探活、调用和下线 | 模型可部署为 HTTP 推理服务并完成调用 |

| F009-resource-monitoring-audit | 实现资源配额、基础监控、日志查询和审计查询 | 关键操作可审计，资源和服务状态可查看 |

## 6. 工作方式

- 每个 Feature 必须在 `docs/features/Fxxx-*` 下建立 plan、TASK、contract、test-plan。
- 每个 Feature 开始前必须确认复用组件、接口契约、权限模型和验收标准。
- 每个 Feature 完成后必须提供测试证据、截图或接口结果、审计记录和已知风险。
- 后端、前端、部署和 AI 组件集成必须保持可重复构建。
- 组件版本、环境变量、密钥、存储桶、镜像仓库不得散落在代码中，应集中在部署配置和环境配置中管理。

## 8. 项目风险

| 风险 | 等级 | 缓解措施 |
| --- | --- | --- |
| 私有化 K8s/GPU/NPU 环境延迟 | 高 | 第 0 阶段锁定环境，准备最小验证集群 |
| 企业 SSO/IAM 对接滞后 | 高 | 先用 Keycloak PoC，保留企业 IAM 适配层 |
| 标注、训练、推理组件集成复杂 | 高 | 每个组件先做 PoC，版本矩阵通过后进入 MVP |
| 数据安全规则不明确 | 中 | 建立待确认清单，MVP 先做最小权限和审计基线 |
| 边端网络不可控 | 中 | agent 设计支持重试、状态回传和离线包 |
| Scope 膨胀 | 高 | 以 MVP 主线验收为准，二期能力单独排期 |

## 9. 待确认事项

- 项目正式启动日期、UAT 时间窗口和上线窗口。
- 真实 BU 试点范围、试点用户和样例数据集。
- 企业 SSO/IAM、镜像仓库、对象存储、K8s 集群是否已有标准环境。
- GPU/NPU 型号、资源数量、资源池隔离要求。
- 数据集和模型的安全等级、留存周期、下载审批规则。
- 边端服务器系统、网络、安全策略和部署方式。
